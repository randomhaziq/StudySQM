## Introduction
Software Quality Assurance (SQA) ensures that quality is built into every phase of the Software Development Lifecycle (SDLC).
It includes processes, standards, and activities that guarantee the software is developed correctly and meets customer expectations.
>SQA ensures “quality by process,” while QC ensures “quality by testing”.

## Why SQA Is Integrated Into Software Development Methodology?
SQA must be built into the SDLC to:
- Prevent defects (not just find them)
- Ensure consistency and adherence to standards
- Improve process reliability
- Reduce project risks
- Increase customer confidence
- Support compliance (ISO, CMMI, audits)
- Ensure traceability from requirements → release

## SQA Activities Across the SDLC
1. Requirements Phase
   - SQA Activities:
     - Requirements review
     - Validation of completeness, consistency, clarity
     - Standards for documenting requirements
     - Traceability matrix starts here
   - Purpose: Reduce requirement defects early.

2. Design Phase
   - SQA Activities:
     - Architecture reviews
     - Design walkthroughs & technical reviews
     - Compliance with design standards
     - Security and performance considerations
   - Purpose: Ensure sound technical structure.

3. Coding / Development Phase
   - SQA Activities:
     - Coding standards
     - Static code analysis (SonarQube)
     - Peer reviews/code inspections
     - Version control and branching policies
   - Purpose: Ensure maintainable, clean, reliable code.

4. Testing Phase
   - SQA Activities:
     - Test planning & test case reviews
     - Coverage analysis
     - Defect management
     - Ensure test process follows standards (IEEE 829 / ISO guidelines)
   - Purpose: Verify functionality, reliability, and performance.

5. Deployment Phase
   - SQA Activities:
     - Release readiness review
     - Configuration management audits
     - UAT compliance checks
     - Documentation verification
   - Purpose: Ensure stable, controlled deployment.

6. Maintenance Phase
   - SQA Activities:
     - Change control
     - Regression testing
     - Impact analysis
     - Post-implementation review
   - Purpose: Maintain reliability and avoid introducing new defects.


## SQA in Agile Methodology
Agile integrates SQA continuously. 
- Key Agile SQA practices:
  - Definition of Done (DoD)
  - Continuous Integration / Continuous Delivery (CI/CD)
  - Automated testing
  - Pair programming
  - Peer code reviews
  - Sprint reviews & retrospectives
  - Story refinement and acceptance criteria
  - Test-driven development (TDD)
- Purpose: Quality is built into every sprint.

## SQA in Waterfall Methodology
- Waterfall uses formal SQA checkpoints:
  - Requirement reviews
  - Design inspections
  - Test document audits
  - Quality gates between phases
- Purpose: Ensures quality before moving to the next phase.

## Tools Supporting SQA in Development Methodology
Common tools:
- Jira / Azure DevOps (process tracking)
- GitHub / GitLab (code reviews, version control)
- Jenkins / GitHub Actions (CI)
- SonarQube (static code quality)
- Selenium / JMeter (test automation)

## Metrics Used to Monitor SQA Across SDLC
These metrics help assess quality at different stages:
- Defect density
- Requirements stability index
- Review effectiveness
- Code complexity
- Test coverage
- MTTR / MTTF
- Build success rate (CI/CD)
- Defect leakage rate

## Benefits of Integrating SQA Into Methodology
- Early defect detection → lower cost
- Better development discipline
- Higher product reliability
- Repeatable and predictable processes
- Compliance with standards (ISO/IEC, CMMI, IEEE)
- Improved communication & documentation
- Supports DevOps and continuous quality
