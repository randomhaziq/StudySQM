# Software Audit

## 1. What is Software Audit?
Software Audit is a systematic process used to evaluate whether software products, development processes, documentation, security controls, and operational procedures comply with established standards and achieve the required quality levels.

It is an essential component of **Software Quality Assurance (SQA)** and **Software Quality Management (SQM)**, helping organizations improve software quality, reduce risks, and ensure compliance with industry best practices.

Keys:
- Software products
- Development processes
- Documentation
- Security controls
- Compliance with standards and regulations

---

# 2. Objectives of Software Audit

## a) Assess Compliance

Ensure that the organization complies with:

- Internal policies
- Industry standards
- Customer requirements
- Government regulations

### Examples

- ISO 9001
- ISO/IEC 25010
- ISO/IEC 27001
- CMMI

---

## b) Evaluate Software Quality

The audit evaluates whether the software meets quality requirements in terms of:

- Reliability
- Security
- Performance
- Maintainability
- Usability

---

## c) Identify Risks

Examples of common risks include:

- Lack of backup procedures
- Weak password policies
- Absence of version control
- Insufficient testing practices

---

# 3. Types of Software Audit

Conducted by the organization's own audit team.

### Purpose

- Internal process improvement
- Preparation for external audits

---

## External Audit

Conducted by independent external parties.

### Examples

- ISO auditors
- Government auditors

---

## Compliance Audit

Evaluates compliance with specific standards or regulations.

### Examples

- ISO 9001
- ISO/IEC 27001

---

## Security Audit

Focuses on information security and cybersecurity practices.

### Areas Reviewed

- Authentication
- Authorization
- Encryption
- Vulnerability management
- Access control

---

## Process Audit

Evaluates software development processes.

### Examples

- Agile
- Scrum
- DevOps
- CI/CD practices

---

# 4. Software Audit Phases

## Phase 1: Audit Planning

Activities include:

- Defining audit scope
- Establishing audit objectives
- Preparing audit schedules

---

## Phase 2: Evidence Collection

Auditors gather evidence from:

- Project documents
- Source code repositories
- Test reports
- Development records

---

## Phase 3: Interviews

Interviews may involve:

- Project Managers
- Software Developers
- Testers
- DevOps Engineers
- System Administrators

---

## Phase 4: Evaluation

Collected evidence is compared against:

- Standards
- Policies
- Procedures
- Best practices

---

## Phase 5: Audit Reporting

The audit report typically includes:

- Findings
- Non-conformities
- Risks
- Recommendations

# 8. Auditing Based on ISO/IEC 25010

ISO/IEC 25010 defines software quality characteristics.

| Quality Characteristic | Description |
|-----------------------|-------------|
| Functional Suitability | Software functions meet user requirements |
| Performance Efficiency | Efficient use of resources and response times |
| Compatibility | Ability to coexist and interact with other systems |
| Usability | Ease of use and user satisfaction |
| Reliability | Ability to perform consistently under specified conditions |
| Security | Protection of information and resources |
| Maintainability | Ease of modification and enhancement |
| Portability | Ability to transfer software across environments |

# 10. Audit Findings

## Major Non-Conformity

A significant issue that may seriously impact the system or organization.

### Examples

- No database backup strategy
- Lack of access control mechanisms
- Critical security vulnerabilities

---

## Minor Non-Conformity

A less severe issue that requires correction.

### Examples

- Incomplete documentation
- Missing approval signatures

---

## Observation

An opportunity for improvement.

### Examples

- Increase automated testing coverage
- Improve deployment documentation

---

# 12. Common Audit Questions

1. Where are system requirements documented?
2. How are requirement changes managed?
3. Is code review performed before merging code?
4. Is automated testing implemented?
5. How is backup managed?
6. How is deployment performed?
7. Is there a rollback plan?
8. Who approves software releases?
9. How are security issues handled?
10. Is an audit trail maintained?
